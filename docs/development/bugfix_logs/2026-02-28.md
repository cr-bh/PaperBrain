## Bug 1: 导入论文后点击"查看论文详情"无法跳转

### 问题描述

在 Auto-Scholar 收藏列表中，将论文导入论文库后，点击出现的"📖 查看论文详情"按钮，页面无反应，无法跳转到论文笔记界面。

### 复现步骤

1. 打开 Auto-Scholar → 收藏列表
2. 点击某篇论文的"📥 导入论文库"
3. 等待导入完成（显示 ✅ 成功）
4. 点击"📖 查看论文详情" → **无反应**

### 根本原因

**Streamlit 按钮嵌套陷阱。**

"查看论文详情"按钮（第 554 行）嵌套在"导入论文库"按钮（第 521 行）的 `if st.button(...)` 代码块内：

```python
# 旧代码（有 Bug）
if st.button("📥 导入论文库"):       # ← 外层按钮
    result = paper_importer.import_arxiv_paper(...)
    if result['success']:
        if st.button("📖 查看论文详情"):  # ← 嵌套按钮
            st.session_state.current_page = 'paper_detail'
            st.rerun()
```

Streamlit 的执行模型：
1. 用户点击"导入论文库" → 脚本重新执行 → 外层 `st.button` 返回 `True` → 导入执行 → 内层"查看详情"按钮**被创建**
2. 用户点击"查看论文详情" → 脚本再次重新执行 → 但此时外层"导入论文库" `st.button` 返回 `False`（因为这次点的不是它） → 外层代码块**不执行** → 内层按钮**从未被创建** → 点击无效

**核心问题**：嵌套在条件块内的按钮，在下一次 rerun 时由于外层条件不满足而消失。

### 解决方案

将导入结果存入 `st.session_state`，把"查看论文详情"按钮移到导入按钮的条件块**外部**：

```python
# 修复后
imported_paper_key = f"imported_paper_{fav.id}"
if st.button("📥 导入论文库"):
    result = paper_importer.import_arxiv_paper(...)
    if result['success']:
        st.session_state[imported_paper_key] = result['paper'].id  # 存入 session

# 按钮在条件块外部，每次 rerun 都能渲染
if imported_paper_key in st.session_state:
    if st.button("📖 查看论文详情"):
        paper_id = st.session_state.pop(imported_paper_key)  # 用后清除
        st.session_state.current_page = 'paper_detail'
        st.session_state.selected_paper_id = paper_id
        st.rerun()
```

### 修改文件

- `ui/auto_scholar.py` — `render_favorite_card()` 函数

### 经验教训

> **Streamlit 规则**：需要在多次 rerun 之间持续存在的按钮，**不能**嵌套在另一个按钮的条件块内。应使用 `st.session_state` 传递状态，并将按钮放在顶层可达的代码路径上。

## Bug 2: 导入论文的结构化笔记内容不全

### 问题描述

导入论文 "UMEM: Unified Memory Extraction and Management Framework for Generalizable Memory" 后，结构化笔记只有前 4 个 section（研究问题、相关工作、现有方案不足、本文贡献），缺少后 5 个 section（具体方法、实验结果、未来工作等）。

### 复现步骤

1. 在 Auto-Scholar 收藏列表导入 UMEM 论文
2. 进入论文笔记页面
3. 结构化笔记只显示 4 个 section，缺少 methodology、results、future_work_paper、future_work_insights、one_sentence_summary

### 根本原因

**LLM 输出 token 数不足导致 JSON 被截断。**

调用链分析：

```
summarizer.summarize_paper()
  → llm.generate_json(prompt, temperature=0.3)   # 未传 max_tokens
    → _call_api(prompt, temperature=0.3)           # max_tokens=None → 使用默认值
      → self._max_tokens = 8192                    # 默认 8192 tokens
```

问题：
1. `generate_json()` 方法签名中**没有** `max_tokens` 参数
2. 结构化总结的 prompt 要求生成 9 个详细 section 的 JSON，输出通常需要 10000-15000 tokens
3. 8192 tokens 不够 → LLM 输出被截断 → JSON 不完整
4. `extract_json_from_text()` 的修复逻辑截断到最后一个完整字段 → 只保留前 4 个 section

### 解决方案

**两处修改：**

1. **`services/llm_service.py`**: 为 `generate_json()` 增加 `max_tokens` 参数

```python
# 修复前
def generate_json(self, prompt, temperature=0.3):
    text = self._call_api(prompt, temperature)
    return extract_json_from_text(text)

# 修复后
def generate_json(self, prompt, temperature=0.3, max_tokens=None):
    text = self._call_api(prompt, temperature, max_tokens)
    return extract_json_from_text(text)
```

2. **`services/summarizer.py`**: 调用时指定 `max_tokens=16384`

```python
# 修复前
summary = self.llm.generate_json(prompt, temperature=0.3)

# 修复后
summary = self.llm.generate_json(prompt, temperature=0.3, max_tokens=16384)
```

### 修改文件

- `services/llm_service.py` — `generate_json()` 方法
- `services/summarizer.py` — `summarize_paper()` 方法

### 经验教训

> **LLM 输出预估**：当 prompt 要求生成大量结构化内容（如多 section 的 JSON）时，必须显式设置足够大的 `max_tokens`。默认值（8192）适合普通问答，但不适合需要详细输出的场景。应在调用点按需指定，而不是依赖全局默认值。
